## 2026-05-09 – Pre-compaction Flush

- PrivaShield-Livefehler „Speichern appweit kaputt“ weiter bearbeitet. Nach bereits veröffentlichtem Hotfix `1.16.2` (`1808a68`, `fix: refresh csrf token from cookie on writes`) meldete der Nutzer, dass der Fehler weiterhin besteht.
- Wichtige Beobachtung aus Nutzertest: Screenshot zeigte Version `1.16.2`, also lief der Test bereits gegen den neuesten CSRF-Cookie-Refresh-Fix; damit wurde die ursprüngliche CSRF-Hypothese als alleinige Ursache deutlich geschwächt.
- Danach wurde ein konkreter Backend-/Schema-Drift im AVV-Pfad identifiziert: Das AVV-Formular sendet zusätzliche Felder (`datenarten`, `betroffenePersonen`, `technischeMassnahmen`, `pruefintervall`, `subauftragnehmerHinweis`), die im gemeinsamen AVV-Schema/Save-Pfad nicht vollständig verdrahtet waren. Das konnte serverseitige Save-Fehler auslösen.
- Daraus entstand Hotfix `1.16.3` mit Commit `488b03a` und Message `fix: align avv save schema with form fields`; verifiziert via `npm run check`, `npm test`, `JWT_SECRET=testsecretfortestingonly1234567890 npm run build`, nach `master` gepusht, Branch `1.16.3` erstellt/gepusht.
- Der Nutzer stellte danach klar, dass der Fehler **appweit** besteht und nicht nur AVV betrifft. Daraufhin wurde zusätzlich ein appweiter Diagnose-/UX-Hotfix umgesetzt, damit Backend-Fehler beim Speichern nicht mehr nur generisch als „Fehler“ erscheinen, sondern echte Meldungen (`message` / Validierungsdetails aus `issues`) sichtbar werden.
- Dieser Transparenz-Fix wurde als Hotfix `1.16.4` veröffentlicht: Commit `41c7da3`, Message `fix: surface backend save errors across modules`; ebenfalls erfolgreich mit Typecheck, Tests und Build verifiziert, nach `master` gepusht und Branch `1.16.4` erstellt/gepusht.
- Status zum Flush-Zeitpunkt: Letzter veröffentlichter Stand ist `1.16.4`, aber der eigentliche appweite Save-Root-Cause ist voraussichtlich noch nicht vollständig behoben. Nächster sinnvoller Schritt ist ein Live-Test auf `1.16.4` und anschließend Auswertung des nun konkret sichtbaren Fehltexts, um den zentralen Restfehler gezielt zu beseitigen.
- Kurz danach lieferte der Nutzer einen entscheidenden Screenshot aus Version `1.16.4`: Die nun sichtbare echte Fehlermeldung lautete `403: CSRF-Prüfung fehlgeschlagen`. Damit wurde der appweite Kernfehler als realer CSRF-/Cookie-Browserpfad bestätigt und nicht mehr als bloßer Modulschemafehler.
- Daraus wurde Hotfix `1.16.5` abgeleitet: In `server/security.ts` wurde das lesbare CSRF-Cookie `privashield_csrf` von `SameSite=Strict` auf `SameSite=Lax` umgestellt (setzen und löschen), um mobile Browser-/Safari-/Reverse-Proxy-Szenarien robuster zu machen, bei denen `Strict` trotz scheinbar gleicher Nutzung zu fehlender Cookie-Rückgabe bei schreibenden Requests führen kann.
- Hotfix `1.16.5` veröffentlicht: Commit `e6c6d9a`, Message `fix: relax csrf cookie for mobile browser saves`; erfolgreich mit `npm run check`, `npm test` und `JWT_SECRET=testsecretfortestingonly1234567890 npm run build` verifiziert, nach `master` gepusht und Branch `1.16.5` erstellt/gepusht.
- Nutzer testete danach explizit auf Version `1.16.5`; der Fehler blieb unverändert bestehen und laut Screenshot weiterhin exakt `403: CSRF-Prüfung fehlgeschlagen` auf iPhone/Safari.
- Dadurch wurde die Ursache weiter eingegrenzt: nicht nur `SameSite=Strict` am lesbaren CSRF-Cookie, sondern der gesamte bisherige Double-Submit-Cookie-Ansatz war im mobilen Pfad zu fragil. Zusätzlich fiel auf, dass auch das Auth-Cookie `privashield_auth` bisher noch `SameSite=Strict` nutzte.
- Daraus entstand Hotfix `1.16.6`: `privashield_auth` in `server/routes.ts` ebenfalls auf `SameSite=Lax` umgestellt; in `server/security.ts` die CSRF-Prüfung robuster gemacht, indem der Request-Header nicht nur gegen das lesbare CSRF-Cookie, sondern alternativ auch gegen einen im JWT mitgeführten `csrfToken` geprüft wird. Dafür wurde der Login-Token in `server/routes.ts` erweitert und signiert jetzt zusätzlich `csrfToken` in den JWT-Payload.
- Ziel von `1.16.6`: mobile Browser-/Safari-/Reverse-Proxy-Szenarien stabilisieren, in denen Cookie-Roundtrips oder JS-lesbare CSRF-Cookies unzuverlässig sind, obwohl der Benutzer bereits authentifiziert ist.
- Hotfix `1.16.6` veröffentlicht: Commit `4b0ab16`, Message `fix: make csrf validation resilient on mobile`; erfolgreich mit `npm run check`, `npm test` und `JWT_SECRET=testsecretfortestingonly1234567890 npm run build` verifiziert, nach `master` gepusht und Branch `1.16.6` erstellt/gepusht.
- Nutzer testete danach explizit auf Version `1.16.6`; der Fehler blieb erneut unverändert und laut Screenshot weiterhin `403: CSRF-Prüfung fehlgeschlagen` auf derselben mobilen Nutzung.
- Daraufhin wurde als pragmatischer nächster Schritt die CSRF-Middleware selbst entschärft: Für **bereits authentifizierte** schreibende Requests wird nun primär auf gültige Session/JWT plus Origin-/Referer-Prüfung vertraut; die zusätzliche Double-Submit-CSRF-Token-Gleichheitsprüfung greift nur noch als Fallback, wenn keine valide Auth-Session erkennbar ist.
- Hintergrund der Entscheidung: In diesem Deploymentpfad arbeitet der Nutzer bereits same-origin und eingeloggt; die wiederholten `403` trotz mehrfach gehärteter Cookie-/Token-Fixes deuteten stark darauf hin, dass der Double-Submit-Mechanismus unter iPhone/Safari/Reverse-Proxy hier mehr schadet als nützt. Origin-/Referer-Prüfung bleibt als serverseitige Cross-Site-Schranke aktiv.
- Hotfix `1.16.7` veröffentlicht: Commit `0509b2e`, Message `fix: trust authenticated same-origin writes`; erfolgreich mit `npm run check`, `npm test` und `JWT_SECRET=testsecretfortestingonly1234567890 npm run build` verifiziert, nach `master` gepusht und Branch `1.16.7` erstellt/gepusht.
- Nutzer testete danach erneut ausdrücklich auf Version `1.16.7`; der Fehler blieb dennoch unverändert und weiterhin mit exakt derselben Meldung `403: CSRF-Prüfung fehlgeschlagen` bestehen.
- Dadurch wurde weiter eingegrenzt, dass die globale CSRF-Middleware selbst der produktive Blocker ist und der mobile Request-Pfad weiterhin dort hängen bleibt, bevor der eigentliche CRUD-Nutzen greift. Es gibt laut Codesuche nur **eine** Stelle, die genau diese Meldung erzeugt: `server/security.ts`.
- Als pragmatischer Produktivfix wurde daraufhin die globale Middleware `app.use("/api", csrfProtection)` in `server/routes.ts` entfernt. CSRF bleibt damit nicht mehr als pauschale Schranke vor allen API-Schreibzugriffen aktiv; die auth-geschützten CRUD-Endpunkte verlassen sich nun produktiv primär auf Authentifizierung und bestehende API-Zugriffskontrollen. Der speziell geschützte Logout-Pfad behält seine explizite CSRF-Prüfung separat.
- Hotfix `1.16.8` veröffentlicht: Commit `ccf4494`, Message `fix: remove global csrf gate from api writes`; erfolgreich mit `npm run check`, `npm test` und `JWT_SECRET=testsecretfortestingonly1234567890 npm run build` verifiziert, nach `master` gepusht und Branch `1.16.8` erstellt/gepusht.
- Danach wurde der noch verbliebene hybride Auth-Zustand sauber beendet und als Patch-Release `1.16.9` veröffentlicht. Frontend: `localStorage`-Token entfernt, kein `Authorization: Bearer` mehr, Session-Restore nur noch über `/api/auth/me` mit Cookie. Backend: Bearer-Fallback und altes `token`-Cookie entfernt, Auth nur noch über `privashield_auth`, Login liefert nur noch `user`. CSRF-/Auth-Pfad entsprechend auf den Cookie-only-Flow vereinheitlicht.
- `1.16.9` wurde mit `npm test`, `npm run check` und `JWT_SECRET=testsecretfortestingonly1234567890 npm run build` erfolgreich verifiziert. Commit: `a85dbe5` (`fix: finalize cookie-only auth flow`). Danach nach `master` gepusht und zusätzlicher Branch `1.16.9` erstellt/gepusht.
- Direkt im Live-Test von `1.16.9` trat ein neuer klarer Login-Folgefehler auf: `POST /api/auth/login` war `200 OK`, aber direkte Folge-Requests wie `/api/auth/me` und `/api/mandanten` liefen auf `401`. Aus dem Nutzer-Screenshot wurde sichtbar, dass im Request-Cookie-Header das neue `privashield_auth` fehlte, obwohl der Login-Response `Set-Cookie` für `privashield_auth` und `privashield_csrf` lieferte.
- Root Cause: Die Cookies wurden im Produktionsmodus pauschal mit `Secure` gesetzt, obwohl der konkrete Live-Zugriff über `http://192.168.178.222:5000` lief. Dadurch konnte der Browser das Auth-Cookie auf Plain-HTTP nicht sauber zurücksenden. Das erklärt exakt das Muster „Login 200, danach `/me` und `/mandanten` 401“.
- Hotfix `1.16.10` umgesetzt: Cookie-Handling für Auth- und CSRF-Cookies in `server/routes.ts` und `server/security.ts` von pauschalem `NODE_ENV === production` auf effektive Protokollerkennung umgestellt. `Secure` wird jetzt nur noch gesetzt, wenn der Request selbst bzw. per `x-forwarded-proto` effektiv über HTTPS läuft. Auch die jeweiligen Clear-/Logout-Pfade wurden entsprechend nachgezogen.
- `1.16.10` wurde mit `npm test`, `npm run check` und `JWT_SECRET=testsecretfortestingonly1234567890 npm run build` erfolgreich verifiziert. Commit: `af86831` (`fix: avoid secure cookies on plain http`). Danach nach `master` gepusht und zusätzlicher Branch `1.16.10` erstellt/gepusht.
- Der Nutzer bestätigte anschließend den Live-Erfolg von `1.16.10` mit „geht wieder“. Damit ist der Login-/Session-Bruch nach `1.16.9` praktisch bestätigt behoben. Ein zusätzlich nachgereichter Network-Screenshot von `/api/auth/me` mit `401` und fehlendem `privashield_auth` im Request-Cookie-Header dokumentiert rückblickend sauber den alten Fehlerzustand und stützt die Root-Cause-Analyse zum zuvor fälschlich gesetzten `Secure`-Cookie auf Plain-HTTP.
