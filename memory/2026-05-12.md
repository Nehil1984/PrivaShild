# 2026-05-12

- PrivaShield Hotfix-Release 1.16.14 umgesetzt und veröffentlicht.
- Ursache: Nach erfolgreichem Login blieb die Oberfläche leer, obwohl Browser-Network-Requests wie `me`, `groups`, `customers`, `assessments`, `findings`, `reports`, `evidence`, `retests`, `users`, `db-config`, `config` und `backups` als `304 Not Modified` zurückkamen. Das sprach gegen einen Auth-Fehler und für ein Frontend-/Response-Handling-Problem.
- Fix in `client/src/lib/queryClient.ts`: JSON-Antwortverarbeitung gehärtet, sodass bodylose oder leere Antworten (`304`, `204`, `205` sowie leerer Body) nicht mehr in einen fehlerhaften JSON-Parse laufen und die UI nach dem Login blockieren.
- Verifikation erfolgreich mit:
  - `npm test`
  - `npm run check`
  - `JWT_SECRET=testsecretfortestingonly1234567890 npm run build`
- Git:
  - Version auf `1.16.14` erhöht
  - Commit: `e8f588d`
  - Message: `fix: handle empty cached api responses`
  - Push auf `master` erfolgreich
  - Branch `1.16.14` erstellt und gepusht

- PrivaShield Hotfix-Release 1.16.15 umgesetzt und veröffentlicht.
- Wunsch: Im Browser-Tab soll nicht die IP-Adresse erscheinen, sondern wie bei den anderen Apps der App-Name.
- Ursache: In `client/index.html` fehlte ein expliziter HTML-`<title>`, weshalb der Browser im Tab statt eines App-Namens nur die Host-/IP-Information anzeigen konnte.
- Fix: `<title>PrivaShield</title>` in `client/index.html` ergänzt.
- Verifikation:
  - `JWT_SECRET=testsecretfortestingonly1234567890 npm run build`
- Git:
  - Version auf `1.16.15` erhöht
  - Commit: `8030a8c`
  - Message: `fix: set app title for browser tabs`
  - Push auf `master` erfolgreich
  - Branch `1.16.15` erstellt und gepusht

- Zusätzlich wichtig für Folgearbeit: Der gemeldete Fehler "nach Login wird nichts geladen" bezog sich nicht auf PrivaShield, sondern auf VulnLedger. Die anfängliche PrivaShield-Analyse war daher eine Repo-Verwechslung; spätere Arbeit wurde auf VulnLedger umgelenkt.

- VulnLedger Folgefixes weitergeführt: Nach 0.3.2 (leere/304 API-Responses robust behandeln) bestand der Fehler weiter. Browser-Hinweise zeigten 304-Verhalten bereits auf dem HTML-Dokument selbst.
- Daher wurde in VulnLedger 0.3.3 die Caching-Strategie der App-Shell im Server angepasst: HTML/Fallback `no-store, no-cache, must-revalidate`, fingerprinted Assets weiter immutable cachebar. Commit `98e9010`, Branch `0.3.3`, Build/Lint erfolgreich.
- Danach zusätzlich VulnLedger 0.3.4 veröffentlicht, um mögliche Nebenwirkungen durch React `StrictMode` beim doppelten Initialisieren des Login-/Session-Ladepfads auszuschließen. `src/main.tsx` rendert jetzt ohne `StrictMode`. Commit `5450934`, Branch `0.3.4`, Push auf `master` und Versionsbranch erfolgt.
- Zusatzänderung VulnLedger 0.3.5: Browsertitel in `index.html` von `vuln-ledger` auf `VulnLedger` angehoben, damit der Tab-Name sauber und produktnah angezeigt wird. Commit `2827ee0`, Push auf `master` erfolgreich.
- Entscheidender Live-Hinweis am 2026-05-12 11:01: Browser-Console zeigte `401 Unauthorized` auf `/api/auth/me` direkt nach dem Login sowie einen Folge-Frontendfehler `Cannot read properties of undefined (reading 'length')`. Damit wurde klar, dass die Session nach Login nicht zuverlässig persistierte. Hotfix 0.3.6: `req.session.save(...)` im Login-Handler ergänzt, sodass die Antwort erst nach gesicherter Session-Speicherung zurückgeht. Commit `591e507` auf `master` gepusht.
- Folge-Hotfix 0.3.7: Reports-/Group-Kontext in `src/App.tsx` gegen fehlende `tenantIds` abgesichert, damit Render-Crashes die gesamte UI nicht schwarz ziehen. Commit `af2091d` auf `master` gepusht.
- Weitere Diagnose zeigte aber weiterhin `401` auf `/api/auth/me` nach erfolgreichem Login. Daher in VulnLedger 0.3.8 den Session-Store selbst von implizitem Default-`MemoryStore` auf `memorystore` umgestellt (`server/index.ts`, Dependency `memorystore`). Ziel: stabilere Session-Persistenz in der Live-Umgebung. Release mit Build/Lint und Push läuft.

- VulnLedger 0.3.9 als grundlegender Auth-Umbau veröffentlicht: Weg von fehleranfälliger `express-session`-Persistenz hin zu einem HttpOnly-Cookie-Token-Ansatz ähnlich PrivaShield. In `server/index.ts` wurden `readNamedCookie`, `setAuthCookie`, `clearAuthCookie` und `getAuthUser` ergänzt; `requireAuth`, `currentUser`, `requireAdmin` und `requireManagerOrAdmin` auf Cookie-Token-Auth umgestellt. Login setzt jetzt `vulnledger_auth` per `jsonwebtoken`, `/api/auth/me` löst den Nutzer aus dem Cookie. Zusätzliche Dependencies: `jsonwebtoken`, `@types/jsonwebtoken`. Verifiziert mit `npm install --include=dev`, `npm run build:full`, `npm run lint`. Git: Commit `c94de61`, Message `fix: move auth to cookie token`, Push auf `master`, Branch `0.3.9` erstellt und gepusht.
- VulnLedger 0.3.10 nach Live-Diagnose mit verbleibendem Frontend-Fehler `Cannot read properties of undefined (reading 'length')` veröffentlicht. In `src/App.tsx` wurden eingehende Session-, User- und Collection-Daten breit normalisiert, insbesondere `user.tenantIds`, `group.tenantIds`, `groups`, `users`, `backups` und weitere Listen, damit `.length`-Zugriffe nicht mehr auf `undefined` laufen. Git: Commit `6c60d9f`, Push auf `master`, Branch `0.3.10` erstellt und gepusht.
- Live-Rückmeldung danach: "es hat geklappt" – die App lud wieder sichtbar korrekt, Login/UI funktionierten, schwarzer Screen war behoben. Es blieben aber noch einzelne `401` auf Endpunkten wie `customers`, `assessments`, `findings` und `retests`.
- VulnLedger 0.3.11 als Restfix veröffentlicht: serverseitige Nutzerauflösung für Lese-Endpunkte gehärtet. Wenn der Nutzer aus dem Auth-Cookie korrekt vorliegt, aber im `listUsers()`-Lookup nicht sauber aufgelöst wird, verwenden die betroffenen Read-Routen jetzt den Cookie-Nutzer als Fallback statt `401 Unauthorized` zu liefern. Verifiziert mit `npm run build:full` und `npm run lint`. Git: Commit `761d63b`, Push auf `master`, Branch `0.3.11` erstellt und gepusht.
